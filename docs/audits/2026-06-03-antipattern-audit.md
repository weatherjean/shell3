## Executive summary

This audit examined the shell3 codebase across error-handling, concurrency, go-idioms, and api-structure dimensions; every finding below survived independent adversarial re-verification against the source. The dominant theme is the chat event pipeline's uniform best-effort `emit()` (`internal/chat/event.go:228-237`), which applies a lossy non-blocking send to lifecycle-terminal events that downstream packages treat as a hard completion barrier — the only findings rated critical/high, because a dropped terminal event can permanently hang or deadlock the `pkg/shell3` embedder. A second cluster of latent concurrency fragility (the OpenAI adapter's cross-goroutine `onEvent` race, the luacfg `TryLock`-based ownership inference, and un-joined turn goroutines touching a closing store) is real but currently masked by single-agent/busy-gate invariants that are documented only in prose. The long tail is low-severity error-context and go-idiom hygiene: discarded errors, unwrapped error returns, and substring/format parsing footguns, plus several issues confined to the `examples/webui` demo.

### [concurrency] Terminal turn_done/error event can be silently dropped, hanging the consumer
- **Location:** `internal/chat/event.go:228-237`  ·  **Scope:** crosscut:concurrency-integration  ·  **Verifier confidence:** high
- **Problem:** emit() does a non-blocking send (`select { case s.events <- ev: default: }`) for ALL events, including the single terminal EventTurnDone/EventError emitted by RunTurn (turn.go:164,173,180). The event channel is buffered (default 256, session.go:43-48). The downstream consumers treat the terminal event as the one signal that the turn finished: pkg/shell3.drain closes the per-Send channel only on Done/Error (shell3.go:176-186), and tui.drainTurn calls app.SetBusy(false) only on EventTurnDone/EventError (interactive.go:341-359).
- **Impact:** If a consumer falls behind (pkg/shell3: drain blocks on the UNBUFFERED `cur <- pub` send at shell3.go:175 because the embedder is draining slowly; TUI: a slow terminal during a heavy token burst), the 256-slot buffer fills and emit() takes the `default:` branch, dropping whatever event is in flight — which may be the terminal turn_done. In pkg/shell3 the per-Send channel is then never closed: the embedder's `for range` over the Send channel hangs forever, and Session.Close()'s `<-s.drainDone` (shell3.go:218) also blocks because drain is wedged on the unbuffered cur send. In the TUI, busy is never cleared: the spinner spins forever and input stays locked. A 'best-effort, droppable' policy is correct for high-volume token events but catastrophic for the lifecycle-terminal events that other packages use as a completion barrier.
- **Fix:** Make the terminal events (and arguably session_end) a guaranteed blocking send while keeping token/reasoning events droppable: e.g. emit() takes a `mustDeliver bool`, and emitTurnDone/emitError/emitSessionEnd use a blocking `s.events <- ev` (still guarded by the recover for the closed-channel case). Alternatively have the consumers detect channel close as an implicit terminal and always clear busy / close cur in a `defer` when the events range ends.
- **Verifier reasoning:** Verified against actual code. emit() (internal/chat/event.go:228-237) performs a non-blocking send `select { case s.events <- ev: default: }` for ALL event kinds, and its own comment (line 224) states "Dropped events are not retried." The terminal lifecycle events go through this same droppable path: RunTurn (turn.go:76) stashes the single end event in `terminalEmit` and invokes it in a defer at turn.go:91-92; the assignments at turn.go:164/173/180 (and panic path at 86) all wrap emitTurnDone/emitError, which call emit(). The channel is buffered at 256 (session.go:43-47, BufSize defaults to 256).  Both downstream consumers treat the terminal event as the sole completion barrier with no fallback on channel-close or range-end: - pkg/shell3 drain (shell3.go:162-188) sends each translated event over an UNBUFFERED `cur <- pub` (line 175) and closes `cur` ONLY on `pub.Kind == Done || Error` (lines 176-186). If the terminal event is dropped, `cur` is never closed, the embedder's range over the Send channel hangs, and Close()'s `<-s.drainDone` (line 218) blocks. - TUI drainTurn (interactive.go:303 `for ev := range ch`, 341-359) calls app.SetBusy(false,...) ONLY on EventTurnDone/EventError; the range ends only at full session teardown (CloseEvents), so a dropped terminal leaves busy stuck for the turn.  Reachability is real, not theoretical: emitAssistantToken (turn.go:349) fires per stream delta — hundreds/thousands per turn — so a stalled consumer (slow embedder blocking the unbuffered cur send, or a slow terminal during a token burst) can fill the 256-slot buffer; since the terminal event is the LAST event emitted, a sustained overflow drops precisely it. Applying a best-effort/droppable policy uniformly to lifecycle-terminal events that other packages use as a guaranteed completion signal is a genuine concurrency antipattern. High severity is appropriate given the hard Close() deadlock in pkg/shell3.

### [concurrency] onEvent callback invoked concurrently from two goroutines (data race)
- **Location:** `internal/adapter/openai/client.go:62, 102, 234, 257-289`  ·  **Scope:** internal/adapter/openai  ·  **Verifier confidence:** high
- **Problem:** RoundTrip starts `go b.scanReasoning(pr, done)` which reads the SSE body via the TeeReader in lockstep with the main Stream loop. scanReasoning calls the `onReasoningDelta` callback (set in Stream to `func(r string){ onEvent(...) }`) on the scanner goroutine, while the main `for stream.Next()` loop simultaneously calls `onEvent` for TextDelta/ToolCall/Usage. Both run concurrently and invoke the same caller-supplied `onEvent` closure.
- **Impact:** The sole caller (chat/turn.go:346 streamOnce) closes over `sb`/`rb strings.Builder`, `toolCalls []llm.ToolCall`, and `usage`, none guarded by a lock. Concurrent onEvent calls from the scan goroutine (ReasoningDelta) and the main loop (TextDelta/ToolCall) produce a genuine data race on those variables — `go test -race` would flag it, and it can corrupt the accumulated reasoning/text/tool-call state or crash.
- **Fix:** Do not call onEvent directly from the scanReasoning goroutine. Instead funnel reasoning deltas through a channel that the main Stream goroutine drains in its select/loop, so all onEvent calls happen on a single goroutine. Document in the llm.Provider/Stream contract that onEvent is invoked from exactly one goroutine.
- **Verifier reasoning:** Verified against the actual code. client.go:62 spawns `go b.scanReasoning(pr, done)` which runs concurrently with the main `for stream.Next()` loop (client.go:257) on the caller's goroutine. scanReasoning invokes `cb(frag)` (line 103) where cb is the closure set at line 234: `func(r string){ onEvent(llm.StreamEvent{ReasoningDelta: r}) }`. The main loop independently calls the same `onEvent` for Usage (261), TextDelta (274), and ToolCall (310). The `b.mu` mutex only guards struct fields and is explicitly released at line 101 BEFORE calling cb at line 103, so it does NOT serialize the two onEvent invocations. The io.Pipe (line 56) + TeeReader (line 57) synchronizes byte transfer (pw.Write blocks until pr is read) but after each read both goroutines independently parse and call onEvent — there is no lockstep that makes the onEvent calls mutually exclusive. The sole caller streamOnce (turn.go:345-360) closes over `sb, rb strings.Builder`, `toolCalls []llm.ToolCall`, and `usage`, mutating all of them inside onEvent with NO lock; strings.Builder.WriteString and slice append are not concurrency-safe, so this is a genuine data race that `go test -race` would flag and that can corrupt accumulated reasoning/text/tool-call state. The llm.Streamer contract (provider.go) does not document single-goroutine invocation, so the caller's assumption is reasonable and the defect is in the adapter. Reachable in practice: the code comments (lines 22-23, 90-96) confirm reasoning fields target OpenRouter/Moonshot/DeepSeek, which interleave reasoning and content/tool deltas in one stream. The claimed location and mechanism are all correct, and the suggested fix (funnel reasoning deltas through a channel drained by the main loop) is sound. Lowered from critical to high: the race is provider-conditional (only fires when reasoning deltas interleave with other deltas, i.e. reasoning-capable non-OpenAI providers) and corrupts per-turn accumulation rather than persistent or security-sensitive state, but it is unquestionably a real concurrency bug.

### [concurrency] Terminal turn_done/error event uses lossy non-blocking emit and can be silently dropped
- **Location:** `internal/chat/event.go:228-237`  ·  **Scope:** internal/chat  ·  **Verifier confidence:** high
- **Problem:** emit() does a non-blocking select send with a default branch, so when the buffered event channel (default cap 256) is full it drops the event. The single terminal event of a turn (emitTurnDone / emitError) goes through this exact path. During a long token stream with a slow consumer the buffer fills with EventAssistantToken events and the terminal event can be the one that gets dropped.
- **Impact:** Both embedding paths key 'turn finished' off the terminal event. In the TUI drainTurn (internal/tui/interactive.go:341-359) only EventTurnDone/EventError call app.SetBusy(false,...); a dropped terminal event leaves the UI busy forever and input is gated while busy. In pkg/shell3 drain (pkg/shell3/shell3.go:176-186) the per-Send channel is closed only on Done/Error, so a dropped terminal event leaves the caller's range-over-channel hung forever. This is a real user-visible hang, not just a lost token.
- **Fix:** Send the terminal event (and other one-shot, non-token events) with a blocking send rather than the lossy non-blocking emit; reserve drop-on-full only for high-volume EventAssistantToken/EventAssistantReasoning. Alternatively guarantee delivery of the terminal event via a separate done signal that the consumer can always observe.
- **Verifier reasoning:** Verified against the actual code. internal/chat/event.go:228-237 defines emit() as a non-blocking `select { case s.events <- ev: default: }` with the comment "emit performs a non-blocking send. Dropped events are not retried." The terminal events go through this exact path: emitTurnDone (lines 215-222) and emitError (lines 189-191) both call emit(), the same function used by the high-volume emitAssistantToken (line 178) and emitAssistantReasoning (line 203). They all share one channel (session.go:47, `events: make(chan Event, opts.BufSize)`), default BufSize 256 (session.go:43-44). There is no ordering or reservation that protects the terminal event from being the one dropped when the buffer is full — it is exactly as droppable as a token.  The claimed consequences are accurate. TUI consumer (internal/tui/interactive.go:341-359): only EventTurnDone and EventError call app.SetBusy(false, nil); a dropped terminal event leaves the UI busy forever. pkg/shell3 consumer (shell3.go:162-188 drain): translate() maps EventTurnDone→Done (line 70-71) and EventError→Error (line 74-75), and the per-Send channel `cur` is closed ONLY when pub.Kind == Done || Error (line 176-186). A dropped terminal event means close(cur) never runs, so a caller ranging over the Send channel hangs forever. Both are genuine permanent-hang failure modes, not lost-token glitches.  This is a real antipattern: a guaranteed-delivery control/terminal event is multiplexed onto the same drop-on-full channel as best-effort high-volume data events. Severity high is justified because the failure mode is a silent permanent hang of the embedding consumer. The trigger requires a consumer slow enough to let 256 buffered events accumulate; the pkg/shell3 path is entirely caller-paced (a slow caller trivially triggers it), and the TUI path can fall behind during fast token streams with heavy rendering. The suggested fix (blocking send for one-shot terminal/control events, drop-on-full reserved for token/reasoning) is sound.

### [concurrency] withIOUnlock uses TryLock to infer caller lock ownership; cross-goroutine Unlock on contention
- **Location:** `internal/luacfg/lua_bash.go:20-33`  ·  **Scope:** crosscut:api-package-boundaries  ·  **Verifier confidence:** high
- **Problem:** withIOUnlock calls c.mu.TryLock() to decide whether it is running inside CallTool's locked section. If TryLock fails it ASSUMES the current goroutine already holds the lock (via CallTool) and calls c.mu.Unlock() then re-Lock(). This conflates 'lock held by me' with 'lock held by anyone'. sync.Mutex has no ownership, so a failed TryLock can also mean another goroutine holds it.
- **Impact:** The documented single-agent invariant is the only thing preventing a bug: if any second goroutine ever calls into the Lua VM (a future HTTP handler, a guard fired concurrently, a test), TryLock fails for the wrong reason and withIOUnlock unlocks a mutex held by a different goroutine — corrupting the other critical section and likely panicking later on a double-unlock. Using mutex acquisition state as a control-flow signal is fragile and unobservable to the race detector.
- **Fix:** Track re-entrancy explicitly (e.g. a per-call boolean / depth counter threaded through CallTool, or pass a 'locked bool' into the Lua-binding closures) instead of probing the mutex. Then Unlock/Lock only when the caller actually held it.
- **Verifier reasoning:** Verified against internal/luacfg/lua_bash.go:20-33. The description is factually exact: withIOUnlock calls c.mu.TryLock() (line 21) and uses the result to infer control flow. On success it treats itself as top-level (DoFile, no caller lock) — Unlock/run/return (lines 22-28). On failure it ASSUMES the current goroutine holds the lock via CallTool and does c.mu.Unlock() then c.mu.Lock() (lines 30-32). This is the classic sync.Mutex-has-no-ownership antipattern: a failed TryLock means "held by anyone," not "held by me." Both lock acquirers exist — CallTool (dispatch.go:109) and runLuaGuard (dispatch.go:48) — and withIOUnlock is reached from Lua bindings (luaBash lua_bash.go:50, luaHTTP lua_http.go:57) invoked inside those locked sections. If a second goroutine ever entered the VM concurrently, goroutine B's TryLock would fail because A holds the lock, and B would Unlock() A's lock, corrupting A's critical section and risking a double-unlock panic. This is invisible to the race detector since it is a logic bug, not a data race on the mutex word. The only thing preventing it today is the documented (not enforced) invariant "luacfg is single-agent; CallTool is never called concurrently," and indeed chat/turn.go:216-229 dispatches tool calls and guards strictly sequentially, so it is not a live bug now — making medium (latent fragility, not active defect) the honest severity rather than high. Note c.L is a single gopher-lua LState that is itself not concurrency-safe, so the mutex's real role is to serialize VM access and withIOUnlock deliberately opens a hole in it during blocking IO. The suggested fix (thread an explicit locked bool / depth counter through CallTool into the bindings instead of probing the mutex) is correct. Location and claim are accurate.

### [concurrency] pkg/shell3 drain blocks indefinitely on the unbuffered Send channel with no ctx escape
- **Location:** `pkg/shell3/shell3.go:169-187`  ·  **Scope:** crosscut:concurrency-integration  ·  **Verifier confidence:** high
- **Problem:** Send creates an unbuffered channel `out := make(chan Event)` (shell3.go:199) and stores it as s.cur. The single drain goroutine forwards every translated event with a blocking `cur <- pub` (shell3.go:175). drain has no select-on-ctx and no escape: if the embedder stops draining the returned channel, drain blocks forever on that send.
- **Impact:** A misbehaving or cancelled embedder that abandons the Send channel wedges the drain goroutine permanently. Session.Close() then deadlocks at `<-s.drainDone` (shell3.go:218) because drainDone is only closed when the drain range exits, which never happens. There is no timeout, ctx, or closed-channel signal that lets drain abandon a stuck send. Combined with finding #1 (dropped terminal event), this means a slow/abandoned consumer can permanently leak the drain goroutine and the whole Session.
- **Fix:** Either document that abandoning a Send channel is a fatal misuse and make Close able to force-unblock drain (e.g. drain selects on a shutdown channel as well as the send), or buffer the Send channel and have drain select between `cur <- pub` and a per-turn ctx.Done()/shutdown signal so an abandoned consumer cannot wedge drain and Close.
- **Verifier reasoning:** Confirmed accurate against the real code. Send creates an unbuffered channel `out := make(chan Event)` (shell3.go:199) stored as s.cur. The single drain goroutine forwards each translated event with a blocking, select-free `cur <- pub` (shell3.go:175). drain's `defer close(s.drainDone)` (shell3.go:163) only fires when the `for ev := range s.sess.Events()` loop exits. Close() (shell3.go:215-218) calls s.sess.CloseEvents() then blocks on `<-s.drainDone`.  The escape claim holds: CloseEvents (internal/chat/session_lifecycle.go:23-25) closes the upstream s.events channel, but a goroutine parked on `cur <- pub` cannot observe that — the range expression only re-evaluates the closed upstream channel after the loop body returns, which it can't because it is stuck mid-iteration on the send. So if the embedder stops reading the Send channel, drain leaks permanently and Close() deadlocks at `<-s.drainDone`. There is no ctx, timeout, or shutdown-channel selected against the send.  Mitigations that cap severity at medium rather than high: the package documents a strict single-turn-drain-to-completion contract (shell3.go:92-94, 190-197), so abandoning a Send channel is explicitly out-of-contract misuse, and the upstream buffer is 256 (NewSession BufSize) so a momentarily-slow consumer is tolerated. But the contract does not cover the realistic path where a consumer cancels its ctx and stops reading: sess.Run honors ctx and stops producing, yet already-emitted/in-flight events plus the terminal Done/Error still hit drain's blocking send, so ctx cancellation does not propagate to drain — a cancelled-and-abandoned consumer can still wedge drain and deadlock Close(). Genuine concurrency robustness gap; the location (shell3.go:169-187) and the line references (199, 175, 218) are correct.

### [concurrency] onReasoningDelta callback can fire after Stream returns
- **Location:** `internal/adapter/openai/client.go:232-241, 99-104`  ·  **Scope:** internal/adapter/openai  ·  **Verifier confidence:** high
- **Problem:** Stream sets `c.tap.onReasoningDelta` and clears it via defer when Stream returns. The scanReasoning goroutine reads `cb := b.onReasoningDelta` under lock then invokes cb. Closing the stream / returning from Stream does not guarantee the scanReasoning goroutine has stopped — WaitReasoning(ctx) returns early on ctx cancellation (line 129-131) without the goroutine finishing.

The goroutine also keeps running until the pipe closes, a transient leak per cancelled request.
- **Fix:** Tie the scan goroutine's lifetime to the request: ensure it is signalled to stop and joined before Stream returns (e.g. close the pipe/read side and wait on done unconditionally, or guard onEvent dispatch behind a per-call generation token checked under the same lock).
- **Verifier reasoning:** Verified against actual code in internal/adapter/openai/client.go. The mechanics in the claim are accurate:  1. The scanReasoning goroutine is launched in RoundTrip (line 62) and its lifetime is NOT tied to Stream's return — nothing joins it unconditionally. 2. Stream sets c.tap.onReasoningDelta under lock (lines 233-235) and clears it via defer (lines 236-240) when it returns. 3. In scanReasoning (lines 99-104), the lock guards ONLY the read of the callback pointer (cb := b.onReasoningDelta), then cb(frag) is invoked OUTSIDE the lock. So the deferred clear cannot stop an already-snapshotted cb from firing. 4. WaitReasoning(ctx) (lines 120-135) returns early on ctx.Done() (line 129-131) WITHOUT waiting on the `done` channel, so on cancellation Stream returns while scanReasoning is still running.  The cb is `func(r string){ onEvent(llm.StreamEvent{ReasoningDelta:r}) }` (line 234). The caller streamOnce (internal/chat/turn.go:346-364) has onEvent write to `rb`, a strings.Builder local to streamOnce (turn.go:352). On ctx cancellation streamOnce returns at turn.go:365-367 while a concurrent scanReasoning goroutine can still invoke cb -> onEvent -> rb.WriteString. Two caveats lowering severity from the claimed high: (a) the description says it writes 'to its strings.Builder' — the sb builder inside scanReasoning (line 69) is goroutine-local and not the racy object; the actual racy write is the caller's rb via onEvent, a minor mischaracterization that does not change the substance. (b) The window is narrow (reasoning delta in-flight at the exact cancellation instant) and the affected data is reasoning text for an already-cancelled turn whose result is typically discarded. It is a real, race-detector-catchable concurrency bug, but practical impact is limited — hence medium rather than high. The suggested fix (join/signal the scan goroutine before Stream returns, or guard onEvent dispatch under the same lock that the clear holds) is appropriate.

### [concurrency] withIOUnlock infers lock ownership via TryLock — latent unlock-of-unowned-mutex panic
- **Location:** `internal/luacfg/lua_bash.go:20-33`  ·  **Scope:** internal/luacfg  ·  **Verifier confidence:** high
- **Problem:** withIOUnlock calls c.mu.TryLock() to decide whether the caller (CallTool/runLuaGuard) already holds c.mu. If TryLock succeeds it assumes 'top-level DoFile, nobody held the lock' and unconditionally calls c.mu.Unlock() without re-locking; if it fails it assumes 'inside CallTool' and does Unlock()+f()+Lock(). The correctness depends entirely on the documented 'single-agent, CallTool never concurrent' invariant.
- **Impact:** TryLock cannot actually distinguish 'I am inside CallTool' from 'another goroutine momentarily released the lock'. If the no-concurrency invariant is ever broken (e.g. someone parallelizes tool dispatch or evaluates guards concurrently, or a future feature drives two LoadedConfig operations at once), TryLock can succeed against a lock a different goroutine owns and then Unlock() it — sync.Mutex.Unlock of a mutex not held by this goroutine panics and corrupts the lock state. This is a fragile, unsound concurrency primitive masquerading as safe.
- **Fix:** Make lock ownership explicit instead of inferred. E.g. have CallTool/runLuaGuard pass a flag or call an unlocked variant of luaBash/httpExec that knows it must release+reacquire, and have the DoFile top-level path run with no lock at all. Removing the TryLock guesswork eliminates the panic risk if the invariant ever changes.
- **Verifier reasoning:** Confirmed accurate at internal/luacfg/lua_bash.go:20-33. withIOUnlock does `locked := c.mu.TryLock()`; if it succeeds it calls `c.mu.Unlock()` + f() and returns (no re-lock), otherwise it does `c.mu.Unlock()` + f() + `c.mu.Lock()`. The field is a plain `sync.Mutex` (luacfg.go:65), which has NO goroutine-ownership concept — so TryLock succeeding only proves the lock was momentarily free, not "the caller (DoFile top-level) does not hold it." The two real callers confirm the intended states: CallTool (dispatch.go:109) and runLuaGuard (dispatch.go:48) both `c.mu.Lock()`/defer Unlock and then drive the Lua VM (which invokes luaBash/luaHTTP on the same goroutine, so TryLock fails -> unlock/relock branch), while Load->DoFile (luacfg.go:82) holds no lock (TryLock succeeds branch). The scheme is therefore sound ONLY under the explicitly documented invariant (line 18: "luacfg is single-agent; CallTool is never called concurrently"). The finding's core technical claim is correct and is a recognized Go antipattern: Go's own sync.TryLock docs warn it should not be used for lock-ownership inference. If the invariant is ever broken (parallel tool dispatch, concurrent guard evaluation, two LoadedConfig-driven operations), TryLock can succeed against a lock another goroutine owns and then Unlock it, producing a "sync: unlock of unlocked mutex" panic and corrupted lock state. This is a latent, fragile concurrency primitive, not an active bug today — it is currently guarded by the invariant — so medium (not high) is the honest severity. The suggested fix (pass explicit ownership flag / unlocked variant rather than inferring via TryLock) is correct. Location and description are factually accurate.

### [concurrency] Close cannot cancel an in-flight turn; turn goroutine can outlive Close and touch a closed store
- **Location:** `pkg/shell3/shell3.go:215-224`  ·  **Scope:** pkg/shell3  ·  **Verifier confidence:** high
- **Problem:** Close calls sess.End, CloseEvents, waits on drainDone, then EndSession and cleanup() (which closes the SQLite store and log). It never cancels the per-Send ctx and the Session holds no cancel func. The turn runs in `go s.sess.Run(ctx, ...)` (line 204) driven solely by the caller-supplied ctx.
- **Impact:** If a caller calls Close while a turn is still streaming (the contract says drain first, but Close has no enforcement or cancellation path), cleanup() closes the store/DB while RunTurn's saveHistory/appendHistory may still write to it — a use-after-close on the DB — and the Run goroutine leaks until the LLM stream ends. Drain exits on CloseEvents, so emits after that are silently dropped (recover). There is no API-level way to stop the turn at teardown.
- **Fix:** Store a context.CancelFunc per active turn (derive a child ctx in Send), and have Close cancel it before CloseEvents/cleanup so the turn unwinds and stops writing to the store before it is closed. Alternatively, wait for the active turn channel to drain before running cleanup.
- **Verifier reasoning:** Verified against the actual code. Close() at shell3.go:215-224 does sess.End("ok") → sess.CloseEvents() → <-drainDone → Store.EndSession → cleanup(). It never cancels the turn and the Session holds no cancel func (struct at lines 95-104 has only mu/cur, no CancelFunc). Send (line 198-206) runs `go s.sess.Run(ctx, ...)` driven solely by the caller-supplied ctx; the only handle is the event channel. So the structural claim is accurate: there is no API-level way for Close to stop an in-flight turn, and the Run goroutine outlives Close until the LLM stream ends — a genuine goroutine leak.  The store-write ordering is also as claimed. chat/session_lifecycle.go:50-59 builds a persist closure (saveHistory→appendHistory→Store.AppendHistory), and RunTurn (turn.go:76-94) runs beforeDone()/persist in a deferred closure that fires AFTER the stream loop completes. agentsetup.go:166 confirms cleanup closes the store (s.Close() → db.Close()) and the log. If a caller calls Close while a turn is still streaming, CloseEvents closes the events chan, drain exits (drainDone closes), and Close proceeds to close the SQLite DB and log while the still-running turn's deferred persist later calls AppendHistory on the now-closed store. That is a real use-after-close call.  Notably, Send's documented single-turn contract (lines 192-197) lists Send/Clear/Rollback/SwitchModel as requiring a prior drain but omits Close, and Close's own doc says nothing about requiring the turn to be drained — so a contract-following caller is not told Close is unsafe mid-turn. The defect is therefore reachable, not purely hypothetical.  Two corrections that lower severity from the claimed high: (1) emit() (event.go:228-237) is non-blocking with default + recover, so the leaked goroutine cannot deadlock on the closed channel — it just drops events; (2) the store is *sql.DB (store.go:16,113,275), so AppendHistory on a closed DB returns sql.ErrConnDone and is logged/swallowed by appendHistory, not a panic or memory corruption. So the real-world harm is a leaked goroutine plus silently dropped history writes (and a Warn to an already-closed log), not a crash or DB corruption. Genuine concurrency antipattern, but medium rather than high given the graceful-failure of database/sql and that it requires a Close-mid-turn the happy path avoids.
